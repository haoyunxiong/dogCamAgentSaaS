# 订单履约交互改写计划

目标：把订单履约页里的“档期查询、创建订单、编辑补资料、发货判断”拆成清晰的运营工作流，减少当前新建/编辑抽屉的信息混杂，同时保留现有 Electron IPC、数据库、顺丰、免押和订单业务逻辑。

## 背景判断

当前页面已经完成一部分 UI 重构：订单列表是表格，订单详情是右侧 Drawer，基础组件也已存在。但“新建订单”仍然把粘贴收货信息、档期查询、创建订单表单放在一个大抽屉里，导致两个不同任务混在一起：

- 查档期：只需要型号、租期、物流方式和大概地址，目的是判断有没有机器、预计发货/送达、报价。
- 创建订单：需要选择设备并记录来源，可先占用档期，客户姓名、电话、详细地址、平台订单号、运单号可以后补。
- 编辑订单：真实高频动作是粘贴收货人姓名、电话、地址并自动识别，然后补齐缺失字段，而不是逐项手填。

本计划只处理订单履约页交互，不重写业务规则。

## 设计原则

- 查档期和创建订单分开呈现，但允许从查档期结果一键复用条件创建订单。
- 创建订单允许“资料未完整”的占用单，但页面必须明确显示缺哪些字段，以及这些缺失会阻止哪些后续动作。
- 订单列表和详情 Drawer 必须持续显示订单状态、履约状态、物流状态、免押状态和资料完整度。
- 编辑订单优先提供“粘贴收货信息并识别”的入口，减少逐字段录入。
- 不把客户电话、姓名作为查档期必要信息；查询只需要城市/区县级别地址，详细地址可选。
- 删除、真实顺丰下单、完结等高风险动作仍使用现有确认机制，不新增突出危险按钮。

## 交互模型

### 1. 档期查询 Drawer

入口从当前“新建订单”拆出为“查档期”或“新建/查档期”里的首选动作。

字段分组：

- 必填：店铺、型号、开始日期、租赁天数或结束日期、物流方式。
- 地址试算必填：省份、城市。
- 可选：区县、详细地址、归还缓冲、手动运输天数。
- 不出现：客户姓名、客户手机号、平台订单号、运单号。

查询结果展示：

- 可用机器列表，默认推荐一台。
- 占用窗口、预计发货、预计送达、租期。
- 租金、押金/免押参考。
- 冲突机器折叠展示。
- 主动作：“复用条件创建订单”。

### 2. 创建订单 Drawer

创建订单不再从空白大表单开始，而是分两种状态：

- 从档期查询结果进入：自动带入店铺、型号、租期、物流方式、地址、推荐设备、费用、押金。
- 直接新建：显示最小必填字段，并提供“先查档期”提示入口。

字段分组：

- 必填用于保存占用单：店铺、型号、设备、租期、来源渠道、来源名称。
- 建议填写：客户姓名、客户手机号、省市区、详细地址、费用、押金。
- 可后补：平台订单号、运单号、物流状态、备注。

保存逻辑：

- 保持调用现有 `window.electronAPI.createOrder`。
- 不降低现有后端档期冲突校验。
- 前端把缺失字段整理成“资料完整度状态”，保存后在订单列表和详情中显示。

### 3. 资料完整度状态

新增前端派生状态，不改数据库：

- `资料完整`：姓名、电话、省市区/详细地址、费用、设备、租期均完整。
- `待补收货`：姓名、电话或详细地址缺失。
- `待补金额`：费用或押金为 0，且没有明确备注。
- `待补平台单号`：平台订单号缺失。
- `待补运单`：已到发货环节但没有运单号。

在列表中作为一个紧凑 Badge 出现，不替代订单状态。详情 Drawer 中展示“缺失字段清单”和下一步动作。

发货动作前：

- 如果缺姓名、电话、详细地址，应提示“收货资料未完整，不能创建真实顺丰运单”。
- 如果只是缺平台订单号，不阻止占用和内部跟进，但显示提醒。

### 4. 编辑订单 Drawer

编辑 Drawer 顶部新增“粘贴收货信息”区：

- 一行 textarea 或紧凑输入框。
- 粘贴后自动解析姓名、电话、省、市、区、详细地址。
- 识别结果以字段预览展示，允许一键应用。
- 支持清空粘贴区，不清空已保存订单字段。

编辑表单重组：

- 第一屏：资料状态、粘贴识别、缺失字段。
- 基础信息：店铺、来源渠道、来源名称、平台订单号。
- 收货信息：姓名、电话、省市区、详细地址。
- 设备租期：型号、设备、租期、费用、押金。
- 物流信息：物流方式、运单号、物流状态。

保存仍调用现有 `window.electronAPI.updateOrder`。

## 实施阶段

### Phase 1：交互结构拆分

文件：

- `electron/renderer/src/views/rental/OrderFulfillment.vue`
- 可新增：`electron/renderer/src/views/rental/order-fulfillment/AvailabilityDrawer.vue`
- 可新增：`electron/renderer/src/views/rental/order-fulfillment/CreateOrderDrawer.vue`

任务：

- 把当前新建抽屉拆成“档期查询”和“创建订单”两个组件。
- 保留现有 `checkScheduleAvailability`、`estimateShipping`、`createOrder` 调用。
- 建立 `availabilityDraft`，用于把查询结果带入创建订单。

验收：

- 可以只填店铺、型号、租期、省市查询档期。
- 查询结果可以一键带入创建订单。
- 直接新建仍可用，但不再把查询和创单挤在同一块大表单中。

### Phase 2：必填/可选/可后补字段治理

文件：

- `OrderFulfillment.vue`
- `CreateOrderDrawer.vue`
- `OrderTable.vue`
- `OrderDetailDrawer.vue`

任务：

- 在创建 Drawer 中明确标识“保存必填”“发货前必填”“可后补”。
- 新增前端派生函数 `getOrderCompletionState(order)`。
- 列表新增资料状态 Badge。
- 详情 Drawer 新增缺失字段清单。

验收：

- 资料不全的订单能保存为占用单。
- 列表能看出缺什么。
- 发货相关操作前能看出是否缺收货资料。

### Phase 3：编辑 Drawer 快速补资料

文件：

- `OrderFulfillment.vue`
- 可新增：`electron/renderer/src/views/rental/order-fulfillment/RecipientPastePanel.vue`
- 可新增：`electron/renderer/src/views/rental/order-fulfillment/EditOrderDrawer.vue`

任务：

- 抽出当前 `parseRecipientInfo`、`parseChinaAddress`、`applyRecipientInfo` 逻辑，复用于新建和编辑。
- 编辑 Drawer 顶部加粘贴识别。
- 粘贴识别后只覆盖识别出的字段，避免误清空人工已填内容。

验收：

- 打开编辑订单后，可以粘贴“姓名 电话 地址”并自动填入。
- 未识别字段不会覆盖原字段。
- 保存后列表和详情 Drawer 同步更新。

### Phase 4：发货前资料门禁

文件：

- `OrderFulfillment.vue`
- `useOrderActions.js`
- `OrderDetailDrawer.vue`

任务：

- 在“顺丰寄件”和批量寄件前检查姓名、电话、详细地址、省市区。
- 对缺资料订单给出可操作提示：打开编辑 Drawer 补资料。
- 不改变顺丰真实下单 API 和现有确认逻辑。

验收：

- 缺收货资料的订单不会直接进入真实顺丰下单。
- 用户能从提示直接跳转到编辑补资料。
- 资料完整的订单保持原有寄件流程。

### Phase 5：视觉和回归验证

验证命令：

```bash
cd electron
npm run build:renderer
npm run dev
```

人工/浏览器验证：

- `/orders` 页面无控制台错误。
- 查档期、复用创建、直接创建、编辑补资料、保存订单、打开详情 Drawer 均可用。
- 订单列表仍显示订单状态、履约状态、免押状态、物流状态。
- 删除仍需要确认。
- 未修改 launcher、Electron main、Python bridge、数据库 schema、IPC 协议、顺丰业务逻辑。

## 明确不做

- 不把订单页改成全新业务系统。
- 不新增大型 UI 框架。
- 不改数据库字段来保存资料完整度，第一版用前端派生状态。
- 不改飞书、手机 H5、顺丰 API、免押 API 的业务行为。
- 不在本阶段做多用户、权限、云端后端拆分。
