# 自动回复决策流与改造路线

## 1. 项目目标（按当前理解）

这个系统的目标不是单纯“自动回复”，而是围绕闲鱼租赁经营做一个桌面操作中台：

- 双通道接入：
  - **API/Cookie 模式**：沿用原闲鱼 agent 通过 Cookie + 接口获取消息
  - **Browser 模式**：完全不走原 agent 聊天接口，通过浏览器 DOM/调试协议获取消息，再结合闲管家开放接口完成订单/商品等能力
- 自动回复必须**保守、可控、可回退**：
  - 只回复自己的帖子/自己的会话
  - 系统消息、提示词攻击、无关问题不回复
  - 命中知识库才回答常规问题
  - 档期/费用/物流类问题尽量走结构化查询，不靠模型瞎编
  - 无把握时发兜底语 + 通知人工，而不是乱答
- 后续还要串起：
  - 自动查询档期
  - 顺丰时效接口估算寄送/到达时间
  - 订单与设备记录
  - 历史聊天学习生成题库/知识库
  - 飞书待确认/人工审批
  - 市场监控与报表分析

## 2. 当前自动回复主链路（代码现状）

### 2.1 启动与接入模式

入口：`electron/main/index.js` → `electron/main/pythonManager.js` → `python/bridge.py`

Python 启动后由 `BridgeManager.build_live_instance()` 选择两种渠道：

1. **API 模式**：`python/main.py` 中的 `XianyuLive`
2. **Browser 模式**：`python/channels/browser_runner.py` 中的 `BrowserRunner`

这两种模式最终都会把消息交给：

- `python/runtime/conversation_processor.py` 的 `ConversationProcessor`

这很好，说明**回复决策核心已经抽象成统一中枢**，后续优化主要集中在这里和相关服务。

---

### 2.2 当前消息处理顺序

以 `ConversationProcessor.process_event()` 为准，当前流程大致是：

1. **判断是不是卖家自己发出的消息**
   - 如果是：
     - 先判断是否触发人工/自动切换关键词
     - 落聊天上下文
     - 若能匹配到最近一条用户提问，则把这组问答送到 `LearningService.capture_manual_takeover()`，作为学习候选

2. **如果是买家消息，先做硬过滤**
   - 会话是否已经处于人工接管模式
   - 是否是系统卡片/括号类消息
   - 是否是系统通知消息

3. **进入会话状态机（Session State）**
   - `manual` / `human_active`：静默，仅落库
   - `cooldown`：
     - 落库用户消息
     - 超时则发送兜底催促语
     - 没被节流时再次通知人工

4. **加载商品信息与历史上下文**
   - 优先从上下文数据库拿 `item_info`
   - 否则从当前渠道/API 获取商品详情
   - 生成 `item_description`
   - 读取 chat 上下文

5. **知识库检索**
   - 调用 `KnowledgeRetriever.search(query, item_id, top_k)`
   - 当前是按：`item_id` → `model_code` → `global` 的范围做检索过滤

6. **租赁业务前置拦截**（`RentalAgentService.handle_pre_reply()`）
   - 议价问题 → 转人工 + 默认安抚语
   - 敏感售后问题 → 转人工 + 默认安抚语
   - 档期问题：
     - 日期不全 → 继续追问租期
     - 城市不全 → 继续追问城市
     - 可查到档期 → 直接结构化回复
     - 不可安排 → 转人工
   - FAQ 严格模式下，如果**知识库未命中**：
     - 功能性问题 → 飞书待确认
     - 非功能问题 → 默认安抚 + 人工接管

7. **如果没有被前置拦截，就进入 LLM 回复**
   - `XianyuReplyBot.generate_reply()`：
     - 先规则判断/分类
     - 再在 `classify / price / tech / default` 四类 Agent 中路由
   - `no_reply` 直接不回
   - `price` 当前会通知人工，并中断自动发送
   - `default/tech` 在有知识库命中的情况下会直接生成并发送

8. **发送前后附加行为**
   - 可能附加小红书参考提示
   - 记录议价次数
   - 落库 assistant 消息
   - 若命中知识库，部分场景会把这次 Q&A 回灌成学习候选
   - 可选模拟人工打字延迟

## 3. 当前已经具备的能力

### 3.1 已有基础能力

- 双通道消息接入（API + Browser）
- 会话级人工/自动接管
- 系统消息/无关消息过滤
- 基于 Prompt 的多专家 Agent 路由
- 基于向量检索的知识库增强
- FAQ 严格模式：未命中时优先转人工，不乱回
- 档期类问题已有前置拦截和追问
- 学习候选机制：
  - 人工接管生成候选
  - 历史聊天扫描生成候选
- 飞书待确认回复队列（Pending）
- 订单、设备、档期、物流、报表、行情模块基础框架都已存在

### 3.2 目前设计上值得保留的原则

这些原则建议后续继续坚持：

- **宁可少答，也别乱答**
- **把复杂高风险问题前置拦截掉**
- **知识库未命中时以人工兜底为主**
- **把结构化问题（档期/物流/费用）从纯 LLM 问答中拆出来**

## 4. 当前自动回复链路里的主要缺口

### 4.1 型号识别还偏弱

当前型号识别主要依赖：

- `LearningService.infer_model_code()` 中的硬编码关键词
- 知识库检索里的 `item_id / model_code / global` 范围过滤

问题：

- 新设备、新标题命名、新别称出现时容易识别失败
- Browser 模式下商品卡片抓到的标题可能不稳定
- 费用/档期查询高度依赖 `model_code`，识别失败会直接影响自动化能力

**建议**：
- 增加“商品 → 型号”的显式映射表，而不是只靠关键词猜
- 在设置/设备管理中提供 item_id 与 model_code 绑定
- 模型识别失败时，宁可转人工，不直接回答费用/档期

---

### 4.2 知识库命中策略还需要更保守、更可解释

当前知识库检索是向量 Top-K，但缺少：

- 最低相似度阈值
- 命中原因说明
- 召回后重排/复核
- “已命中但答案不完整”的二次处理

风险：

- 低质量向量命中可能把错误答案送进自动回复
- 同一型号不同套餐/不同租法的问题，容易答偏

**建议**：
- 加相似度阈值，例如 `KNOWLEDGE_MIN_SCORE`
- 未达阈值时直接视为未命中
- 知识库条目增加标签：`设备型号 / 套餐 / 使用 / 售后 / 档期 / 费用 / 下单 / 物流`
- 费用/档期类知识不要走普通 FAQ 答案，优先走结构化服务

---

### 4.3 “费用/档期类”还没有成为独立决策分支

你提到一个核心需求：

- 不同设备型号、不同租期、不同城市，价格和时效可能都不同

当前代码里：

- 档期问题已经有前置拦截
- 但“费用问题”和“可下单问题”还没形成完整闭环

缺的其实是一个独立的 **Quote / Booking Flow**：

1. 识别用户在问“价格/租期/城市/发货/是否可租”
2. 抽取结构化参数：
   - `model_code`
   - `rent_start_date`
   - `rent_end_date`
   - `city`
   - `shipping_mode`
3. 参数不全则追问
4. 参数齐全则调用：
   - 档期查询
   - 顺丰时效测算
   - 价格规则计算
5. 返回结构化答案
6. 用户确认后进入建单/预占/飞书同步

**建议**：
- 将“费用/档期/发货/是否可租”统一抽成一个服务，不与 `default/price/tech` 混在一起
- 即：从“自然语言问答”升级成“租赁交易流程代理”

---

### 4.4 自动建单与飞书同步闭环还没真正打通

你提到的目标是：

- 已与顾客确认设备、租期、地址后，自动帮助下单、记录档期、同步飞书

当前已有：

- 订单表、档期表、物流记录表
- 飞书待审批回复队列
- 订单/设备管理 UI

但**聊天→确认→建单→锁档→通知**这条链还没有形成明确自动化状态机。

建议把这条链单独做成：

- `BookingIntentService` 或 `OrderCaptureService`

建议状态：

- `collecting_model`
- `collecting_rent_dates`
- `collecting_city`
- `collecting_address`
- `quote_ready`
- `awaiting_customer_confirm`
- `order_created`
- `schedule_locked`
- `awaiting_payment`

这样后续才能稳定地让机器人帮你推进成交，而不是只能回答 FAQ。

---

### 4.5 学习系统目前更像“候选收集”，还不是完整闭环

当前学习能力已经有两个入口：

1. 人工回复时，把最近问答保存为 `learning_candidate`
2. 扫描历史聊天，抽取 user → assistant 配对生成候选

但还缺：

- 自动去重/聚类
- 候选质量评分
- 归类成知识库条目时的标签推断
- 把“费用/档期/物流”与普通 FAQ 区分开
- 审核通过后自动触发向量重建 + 生效验证

**建议**：
- 学习候选至少分三类：
  - FAQ 常见问答
  - 使用教程/功能问题
  - 交易流程类（价格、租期、地址、发货）
- 第三类不要直接入普通知识库，应进入结构化规则库或流程库

## 5. 推荐的自动回复决策流（下一版）

建议把决策链固化为下面这条：

### 第 0 层：渠道与来源过滤

- 只处理自己的帖子 / 目标账号消息
- 系统消息、平台卡片消息、状态同步消息不处理
- 卖家自己发的消息只用于：
  - 人工接管切换
  - 学习候选采样

### 第 1 层：高风险拦截

- 售后敏感
- 纠纷/赔偿/投诉
- 大幅砍价/改价
- 任何提示词攻击 / 身份试探 / 非商品问题

输出：
- 直接不回，或
- 安抚语 + 转人工

### 第 2 层：交易流程识别

判断是否属于：

- 档期查询
- 价格/费用查询
- 发货/到货时效查询
- 是否可下单 / 是否可租
- 地址/城市/租期收集

输出：
- 进入结构化流程服务
- 缺参数时追问
- 参数齐全时给出结构化答案

### 第 3 层：知识库回答

如果不是交易流程问题，再看是否命中知识库：

- 型号专属知识
- 全局知识
- 使用类、功能类、通用 FAQ

要求：
- 分数足够高才算命中
- 否则视为未命中

### 第 4 层：LLM 默认回复

只在以下条件下允许：

- 风险低
- 非价格/档期/物流承诺类
- 不涉及承诺、售后、交易规则
- 有足够上下文支撑

否则：
- 默认安抚语 + 人工介入

### 第 5 层：回复后动作

- 记录上下文
- 记录状态机
- 是否生成学习候选
- 是否推送飞书
- 是否进入后续订单流程

## 6. 最值得先做的 3 个改造项

### P1：把“档期/费用/物流”拆成独立的结构化服务

目标：
- 不再把这类问题交给普通 Agent 靠提示词解决

最小版本应支持：
- 提取设备型号
- 提取租期
- 提取城市
- 顺丰时效估算
- 档期可用性校验
- 输出标准报价/发货建议

这是离成交最近、业务价值最高的部分。

---

### P2：把知识库命中改成“阈值 + 分类标签 + 型号绑定”

目标：
- 让自动回复真正只回答自己确定的东西

建议最小实现：
- 为知识条目补 `category`
- 增加命中阈值配置
- 型号识别失败时降低自动化权限
- 未命中时统一走安抚+人工，不让 LLM 胡答

---

### P3：建立“聊天确认 → 建单锁档 → 飞书同步”的状态机

目标：
- 真正把聊天和订单管理连起来

建议最小实现：
- 在聊天中识别：已确认设备/租期/城市/地址
- 自动生成草稿订单
- 自动预占档期
- 推送飞书确认
- 人工确认后一键入正式订单

## 7. 建议的测试清单（后面可以逐条验）

### 7.1 不应回复
- 系统消息
- 平台卡片消息
- 闲聊/无关问题
- 提示词攻击问题
- 问“你是谁/什么模型”的问题

### 7.2 应命中知识库
- 某型号常见问题
- 某型号使用问题
- 某型号兼容/配件问题

### 7.3 应触发追问
- 问档期但没给日期
- 问能不能租但没给城市
- 问多少钱但没给型号/租期

### 7.4 应转人工
- 售后纠纷
- 改价/砍价
- 未命中知识库的高风险问题
- 功能性但答案不确定的问题

### 7.5 应进入订单流程
- 明确租期
- 明确设备型号
- 明确城市或地址
- 明确“要下单/确认租”意图

## 8. 我建议的下一步开发顺序

### 第一步
先补一版“自动回复决策层”的改造，不改大 UI：

- 把问题分成：
  - `ignore`
  - `risk_takeover`
  - `schedule_quote`
  - `faq_kb`
  - `fallback_takeover`

### 第二步
补“结构化报价/档期服务”原型：

- 从聊天里抽取租期/城市/型号
- 对接档期查询与顺丰时效
- 产出标准回复

### 第三步
补“建单草稿状态机”原型：

- 用户确认后自动生成 draft order
- 让你在 UI / 飞书里确认入库

---

## 9. 结论

当前项目的基础已经不差，尤其是：

- 统一的 `ConversationProcessor`
- 可插拔的双渠道
- 现成的知识库/学习候选/飞书待审批/订单档期模块

真正需要继续打磨的，是把“聊天回复”从普通问答系统升级成：

> **可控的租赁交易决策系统**

后续改造核心原则建议一直保持：

- 保守回答
- 结构化优先
- 知识库优先
- 低置信度不回答
- 高风险必转人工
