## 4. 产品设计原则

### 4.1 产品定位

小狗相机助手商户版是面向租赁商家的运营标准化系统，用于规范和提效商户内部的询单、订单、档期、设备、免押/押金、物流、归还、验机、员工协作和经营数据管理流程。

### 4.2 体验关键词

```text
专业、可信、克制、清爽、高效、数据化、SaaS 感
```

### 4.3 设计取向

1. 面向日常高频运营，不做营销页。
2. 信息密度要高，但必须可扫读。
3. 核心业务状态必须显眼，不能被装饰元素稀释。
4. 小狗 IP 可用于 logo、登录页、空状态、引导提示，不进入核心业务页面的视觉主导。
5. 闲鱼只作为 legacy channel / 渠道来源之一，不作为新产品主入口。
6. 手机端是移动运营工作台，PC 端是完整管理后台。

## 5. 手机端与 PC 端任务分工

| 维度 | 手机端 | PC 端 |
|---|---|---|
| 产品定位 | 移动运营工作台 | 完整管理后台 |
| 使用场景 | 在店里、仓库、发货现场、临时查单、快速处理待办 | 坐在电脑前批量处理、核对数据、配置规则、分析经营 |
| 信息形态 | 卡片、任务列表、快捷入口、轻筛选、轻确认 | 表格、复杂筛选、批量操作、侧栏详情、报表、设置 |
| 操作特点 | 快速判断、快速确认、快速进入详情 | 批量管理、精细编辑、复杂核对、导入导出预留 |
| 不做什么 | 不做复杂大表格、不做完整后台配置、不做深度报表 | 不做过度移动化卡片堆叠、不牺牲批量效率 |

## 6. 手机端页面范围

Phase 02A 手机端核心页面归并为 7 个设计对象：

1. 登录页
2. 工作台
3. 订单中心
4. 订单详情 / 订单处理页
5. 档期中心 / 档期查询
6. 设备中心 / 设备快查
7. 我的 / 商家中心

说明：

1. 订单中心与订单详情可以拆成两个设计任务。
2. Phase 02C 高保真原型阶段，优先按“订单中心 + 订单详情抽屉 / 详情页”作为一个核心模块处理，避免扩展成过多页面。
3. 登录页只作为设计输入，不代表 Phase 02 实现登录系统。登录、商户、门店、员工、权限、数据隔离归属 Phase 03。

## 7. 手机端页面设计输入

### 7.1 登录页

| 项目 | 内容 |
|---|---|
| 主要用户 | 老板 / 超级管理员、店长、运营、发货员、财务 |
| 第一眼要解决的问题 | 用户知道自己正在进入“小狗相机助手商户版”，并能快速完成账号登录 |
| 首屏最高优先级 | 产品标识、登录表单、主登录按钮、登录异常提示位置 |
| 核心操作 | 输入账号、输入密码或验证码、登录、切换登录方式、查看登录失败原因 |
| 默认展示字段 | 产品名称、账号输入、密码 / 验证码输入、登录按钮、记住登录状态入口、轻量帮助入口 |
| 不默认展示字段 | 商户初始化、权限配置、员工管理、详细服务条款、配置中心、CredentialService 说明 |
| 详情 / Drawer / Bottom Sheet | 忘记密码、切换商户、登录失败帮助可使用 Bottom Sheet 或轻量页；Phase 03 再确定真实鉴权逻辑 |
| 状态 | 默认、加载、错误、权限不足、异常 |

设计备注：

1. 登录页可以轻量使用小狗 IP。
2. 不做消费级可爱 App 风格，仍保持专业可信。
3. 不在 Phase 02A 定义真实认证流程。

### 7.2 工作台

| 项目 | 内容 |
|---|---|
| 主要用户 | 店长、运营、发货员、老板 / 超级管理员 |
| 第一眼要解决的问题 | 今天最该处理什么，是否有逾期、异常、待发货、待归还、待审核 |
| 首屏最高优先级 | 今日待办、异常提醒、关键数量、快捷入口 |
| 核心操作 | 进入待办、处理发货、处理归还、查看逾期、进入免押审核、查档期、设备快查 |
| 默认展示字段 | 今日待发货数、待归还数、逾期数、待免押审核数、异常订单数、快捷入口、最近任务 |
| 不默认展示字段 | 完整经营报表、复杂财务明细、全部订单表格、权限配置、历史日志 |
| 详情 / Drawer / Bottom Sheet | 待办列表可进入详情页；快捷筛选和轻确认可使用 Bottom Sheet |
| 状态 | 默认、空状态、加载、错误、权限不足、异常、逾期、待处理 |

首屏结构优先级：

1. 今日待办与异常提醒。
2. 高频快捷入口：查档期、订单处理、发货确认、归还提醒、免押审核、设备快查。
3. 轻量运营概览：今日订单、进行中、待归还、逾期。
4. 最近处理记录或提醒。

### 7.3 订单中心

| 项目 | 内容 |
|---|---|
| 主要用户 | 运营、店长、发货员 |
| 第一眼要解决的问题 | 哪些订单需要我现在处理，它们分别处于什么状态 |
| 首屏最高优先级 | 订单状态筛选、搜索、待处理订单卡片 |
| 核心操作 | 搜索订单、筛选状态、进入订单详情、快速处理待发货 / 待归还 / 逾期 |
| 默认展示字段 | 订单号、客户名、设备型号、租期、订单状态、渠道、金额摘要、下一个动作、风险 / 异常标签 |
| 不默认展示字段 | 完整费用明细、全部操作日志、完整客户资料、详细物流轨迹、完整免押记录 |
| 详情 / Drawer / Bottom Sheet | 筛选条件使用 Bottom Sheet；订单完整信息进入详情页；轻量操作确认使用 Bottom Sheet |
| 状态 | 默认、空状态、加载、错误、权限不足、异常、逾期、待处理 |

订单卡片必须支持的高频状态：

1. 待确认
2. 待免押 / 待押金
3. 待发货
4. 运输中
5. 租赁中
6. 待归还
7. 逾期
8. 待验机
9. 已完成
10. 异常

### 7.4 订单详情 / 订单处理页

| 项目 | 内容 |
|---|---|
| 主要用户 | 运营、店长、发货员、财务 |
| 第一眼要解决的问题 | 这张订单当前最重要的动作是什么，是否有风险或阻塞 |
| 首屏最高优先级 | 订单状态、客户与设备、租期、下一步动作、异常提示 |
| 核心操作 | 确认订单、处理免押 / 押金、发货、查看物流、确认归还、验机、完结订单、联系客户 |
| 默认展示字段 | 订单状态、客户、设备、租期、取还方式、金额摘要、免押 / 押金状态、物流状态、下一步动作 |
| 不默认展示字段 | 全量操作日志、完整财务流水、全部沟通记录、底层渠道字段、legacy Xianyu 技术字段 |
| 详情 / Drawer / Bottom Sheet | 手机端以详情页为主；操作确认、状态切换、物流录入、验机结果可使用 Bottom Sheet |
| 状态 | 默认、加载、错误、权限不足、异常、逾期、待处理 |

信息分层建议：

1. 首屏：状态、风险、下一步动作。
2. 详情区：客户、设备、租期、金额、免押、物流、归还。
3. 折叠区：操作记录、沟通记录、费用明细、附件。
4. Bottom Sheet：发货确认、归还确认、验机结果、异常标记。

### 7.5 档期中心 / 档期查询

| 项目 | 内容 |
|---|---|
| 主要用户 | 运营、店长 |
| 第一眼要解决的问题 | 某设备 / 型号在某日期是否可租，有没有冲突 |
| 首屏最高优先级 | 日期选择、型号 / 设备选择、可租结果、冲突提醒 |
| 核心操作 | 选择日期、选择设备型号、查看可租数量、查看占用详情、复制 / 发送档期方案 |
| 默认展示字段 | 日期范围、设备型号、可租数量、已占用数量、冲突状态、最近占用订单 |
| 不默认展示字段 | 完整订单详情、设备采购信息、历史长期占用记录、复杂算法解释 |
| 详情 / Drawer / Bottom Sheet | 日期选择、型号筛选使用 Bottom Sheet；占用详情进入订单详情 |
| 状态 | 默认、空状态、加载、错误、权限不足、异常、逾期、待处理 |

设计备注：

1. 档期块必须能表达“可租、已占用、冲突、缓冲、待归还、逾期影响”。
2. 手机端优先解决接单时的快速判断，不做完整排班后台。

### 7.6 设备中心 / 设备快查

| 项目 | 内容 |
|---|---|
| 主要用户 | 发货员、店长、运营 |
| 第一眼要解决的问题 | 这台设备现在在哪里，能不能租，是否异常 |
| 首屏最高优先级 | 搜索 / 扫码入口、设备状态、可租情况、当前 / 下一订单 |
| 核心操作 | 搜索设备编号、扫码预留、查看设备状态、进入设备详情、查看关联订单 |
| 默认展示字段 | 设备编号、型号、状态、所在位置、当前订单、下一档期、异常标签 |
| 不默认展示字段 | 采购成本、折旧、完整维修历史、全部配件明细、财务 ROI |
| 详情 / Drawer / Bottom Sheet | 筛选使用 Bottom Sheet；设备详情页展示配件、维修、历史订单等 |
| 状态 | 默认、空状态、加载、错误、权限不足、异常、逾期、待处理 |

设备状态至少需要覆盖：

1. 可租
2. 已预订
3. 租赁中
4. 待发货
5. 待归还
6. 待验机
7. 维修中
8. 不可租
9. 丢失 / 异常

### 7.7 我的 / 商家中心

## 11. 默认展示与非默认展示规则

### 11.1 默认展示规则

默认展示的信息必须满足至少一个条件：

1. 决定用户下一步动作。
2. 影响履约风险。
3. 影响资金安全。
4. 影响设备可用性。
5. 是高频搜索 / 筛选依据。
6. 是当前角色每天都会看的字段。

### 11.2 不默认展示规则

以下信息不进入列表首屏：

1. 完整操作日志。

4. 复杂配置项。
5. 深层财务计算明细。
6. 历史归档数据。
7. Phase 03 之后才定义的权限和配置实现。
8. Phase 08 未来协作网络字段。

## 13. 真实产品参考方向

本项目可参考以下真实产品方向，但只借鉴信息密度、组件组织和管理后台体验，不复制视觉细节：

1. Shopify Admin：商品、订单、客户、设置的后台信息架构。
2. Stripe Dashboard：状态、资金、风险、开发者级清晰度。
3. Linear：高密度列表、状态、筛选和快捷操作。
4. Notion：克制的视觉系统和低干扰信息组织。
5. 飞书 / Lark 管理后台：国内 B 端后台、表格、权限、组织信息的表达方式。

设计时应避免：

1. 重渐变。
2. 玻璃拟态。
3. 霓虹色。
4. 过度卡通化。
5. 营销页式大 Hero。
6. 只为好看而牺牲操作效率。

2. OrderTable 能否在 PC 端承载状态、免押、物流、金额、负责人、渠道。

4. DeviceCard / DeviceTable 能否表达设备编号、型号、状态、位置、当前 / 下一订单。
5. StatusBadge 是否覆盖订单、设备、免押、物流、财务的语义状态。

### 15.3 Phase 02C 高保真优先页面

Phase 02C 高保真原型建议优先做：

1. 手机端：工作台、订单中心 + 订单详情、档期中心、设备中心、我的 / 商家中心。
2. PC 端：工作台、订单中心、档期中心、设备中心、设置中心。

PC 端其他页面作为后续扩展输入，不要求 Phase 02C 一次性全部高保真。
