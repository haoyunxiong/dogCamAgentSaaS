# Phase 02 风险与注意事项：设计系统与 Figma 基础组件

## 主要风险

- 阶段范围过大，导致 Codex 一次性改太多。
- 未先计划就执行。
- 破坏当前可用业务流程。
- 新旧产品定位混乱，继续围绕闲鱼做核心设计。
- 没有及时更新 artifacts 和阶段状态。

## 风险控制

- 每次只做一个小任务。
- 高风险任务先备份。
- 数据库相关改动必须有回滚方案。
- UI 改造必须复用 Design System。
- 每次执行后做验证报告。

## 2026-06-14 新增风险：旧系统改造绑架新体验

当前发现继续在旧系统页面上改造 UI，可能导致新产品体验被旧结构限制，最终变成旧 UI 美化版。

控制方式：

- 暂停 Phase 02C 真实系统 UI 改造。
- 不继续重构旧系统业务页面。
- 先创建独立 UI-V2 Demo 验证新体验。
- Demo 只用 mock 数据。
- Demo 不接旧系统、不接数据库、不接 Electron、不接真实 API。
- Demo 定稿后，再规划旧系统能力迁移。
