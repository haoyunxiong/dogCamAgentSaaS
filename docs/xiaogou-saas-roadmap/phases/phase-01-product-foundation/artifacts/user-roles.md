# Phase 01 用户角色确认

确认日期：2026-06-12

## 核心角色

Phase 01 固定以下核心角色。

| 角色 | 产品定位 | 主要关注点 | 典型任务 |
|---|---|---|---|
| 老板 / 超级管理员 | 经营负责人和最高权限使用者 | 收入、设备收益、订单异常、员工效率、整体经营 | 看经营概览、看报表、处理异常、配置关键规则 |
| 店长 | 门店日常运营负责人 | 今日待办、订单流转、人员分工、设备调度 | 分配任务、盯订单流转、协调设备、处理门店异常 |
| 运营 | 询单与订单前置处理人员 | 询单、查档期、报价、创建订单、客户沟通 | 查档期、发档期、报价、建单、跟进客户 |
| 发货员 | 履约执行人员 | 发货、物流、归还、验机、设备状态 | 打包发货、记录运单、跟进物流、归还验机、更新设备状态 |
| 财务 | 资金和结算负责人 | 租金、押金、退款、结算、报表 | 查收款、核押金、处理退款、看费用和结算 |

## 使用场景角色

| 角色 | Phase 01 处理方式 | 后续阶段 |
|---|---|---|
| 新员工 / 客服运营 | 作为 SOP 使用场景角色记录，不作为 Phase 01 正式权限角色 | Phase 05 员工 SOP 与新员工上手系统 |

## 权限边界

Phase 01 不定义正式权限矩阵，不开发登录、员工、权限或数据隔离。

正式权限体系放到 Phase 03：

```text
登录、商户、门店、员工、权限、数据隔离
```

## 角色设计原则

1. 角色以租赁业务分工为中心，不以闲鱼客服流为中心。
2. 老板和店长关注经营与异常，运营和发货员关注高频动作，财务关注资金口径。
3. 新员工场景先用于 SOP 和培训设计，不提前实现权限模型。
4. 后续权限设计必须支持 merchant_id / store_id / staff / role / permission 的 SaaS 化方向。
