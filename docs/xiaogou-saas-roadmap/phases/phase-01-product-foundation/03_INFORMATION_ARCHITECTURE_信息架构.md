# Phase 01 信息架构：产品定位与系统骨架固定

更新日期：2026-06-12

## 本文件用途

本文件用于固定 Phase 01 的产品骨架输入，包括：

- 产品定位
- 用户角色
- 手机端导航
- PC 端导航
- 核心业务流程
- 页面清单
- 模块边界
- legacy channel 边界

本文件只定义信息架构，不代表已经开发页面、设计 Figma 或修改数据库。

## 产品定位

产品对外名称暂定为：

```text
小狗相机助手
```

产品版本 / 后台名称为：

```text
小狗相机助手商户版
```

产品定位：

```text
面向租赁商家的运营标准化系统，用于规范和提效商户内部的询单、订单、档期、设备、免押/押金、物流、归还、验机、员工协作和经营数据管理流程。
```

当前阶段不再把产品定位为“闲鱼助手”或“闲鱼自动回复工具”。闲鱼只作为 legacy channel / 订单来源渠道之一。

## 行业聚焦范围

Phase 01 先聚焦：

```text
相机、运动相机、CCD、摄影器材、数码设备、演唱会/旅游拍摄设备租赁商家。
```

未来可扩展到其他设备租赁行业，但当前阶段不把产品泛化成所有租赁行业。

## 短期目标

短期目标不是商户之间调货，也不是二手交易平台。

短期目标是：

1. 把当前内部租赁业务流程规范化。
2. 让新员工可以快速上手。
3. 让订单、档期、设备、物流、免押、报表形成标准系统。
4. 让后续多商户 SaaS 化有结构基础。
5. 保留未来商户协作网络的架构预留，但不进入当前阶段实现。

## 用户角色

Phase 01 固定以下核心角色：

| 角色 | 定位 | 关注点 |
|---|---|---|
| 老板 / 超级管理员 | 经营负责人和最高权限使用者 | 收入、设备收益、异常订单、员工效率、整体经营 |
| 店长 | 门店日常运营负责人 | 今日待办、订单流转、人员分工、设备调度 |
| 运营 | 询单与订单前置处理人员 | 询单、查档期、报价、创建订单、客户沟通 |
| 发货员 | 履约执行人员 | 发货、物流、归还、验机、设备状态 |
| 财务 | 资金和结算负责人 | 租金、押金、退款、结算、报表 |

补充使用场景角色：

| 角色 | 处理方式 |
|---|---|
| 新员工 / 客服运营 | 作为 SOP 使用场景角色，不作为 Phase 01 正式权限角色 |

正式权限体系放到 Phase 03：登录、商户、门店、员工、权限、数据隔离阶段处理。

## 手机端信息架构

手机端定位：

```text
移动运营工作台
```

手机端底部导航固定为：

1. 工作台
2. 订单
3. 档期
4. 设备
5. 我的

手机端重点处理：

1. 今日待办。
2. 快速查档期。
3. 快速处理订单。
4. 发货确认。
5. 归还提醒。
6. 免押审核入口。
7. 设备快查。
8. 员工日常任务。

手机端边界：

- 免押不放入底部主导航，作为工作台快捷入口、订单详情入口、PC 独立模块处理。
- 询单不放入底部主导航，可放入工作台快捷入口或订单流程中。
- 手机端不是 PC 页面缩小版，只承载高频轻操作。

## 核心业务流程

Phase 01 固定核心业务流程为：

```text
询单
→ 查档期
→ 报价 / 发档期
→ 创建订单
→ 创建免押 / 押金
→ 发货
→ 物流跟踪
→ 到期提醒
→ 归还验机
→ 完结订单
→ 数据沉淀 / 报表分析
```

Phase 01 只固定流程，不实现状态机，不修改代码。

## Legacy Channel 边界

以下能力统一降级为 legacy / channel / 待确认，不作为新产品主架构：

1. 闲鱼自动回复作为产品核心。
2. 闲鱼账号连接作为主入口。
3. 闲鱼 WebSocket / cookies / token 作为核心业务链路。
4. 浏览器 IM 扫描。
5. 闲管家开放平台。
6. 商品 item_id / chat_id / session_id 强绑定。
7. 自动回复 Agent。
8. 人工接管。
9. 消息审批。
10. 学习审核。
11. 闲鱼市场监控。
12. 闲鱼帖子审核。
13. XianyuAgentPro / xianyu-auto-agent / xianyu_agent 等旧命名。
14. xianyu.deposit* localStorage key。

处理原则：

1. 不删除。
2. 不重构。
3. 不进入新产品主导航。
4. 不作为新架构核心。
5. 只在文档中标记为 legacy / 渠道化候选 / 待确认。
6. 未来如果仍有业务价值，再抽象为渠道能力或客户沟通辅助。
