# 小狗相机助手商户版：产品规划、阶段路线图与 Codex + Figma 改造执行方案

> 文档版本：V1.0  
> 适用对象：产品规划、Figma 设计、Codex 改造当前项目、后续开发排期  
> 当前定位：从“内部闲鱼助手”升级为“面向租赁商家的运营标准化系统”  
> 长期方向：租赁商户 SaaS 系统 + 商户协作网络  

---

## 0. 这份文档的用途

这份文档不是单纯的 UI 美化需求，也不是某一个页面的功能清单，而是用于指导当前项目从“自己使用的后台助手”逐步升级为“可对外提供给租赁商户使用的 SaaS 系统”。

它主要解决 5 个问题：

1. 明确产品不再围绕“闲鱼自动回复”设计，而是围绕“租赁商户经营流程标准化”设计。
2. 明确未来方向，但当前阶段不盲目做商户调货、代发和交易闭环。
3. 固定系统结构、导航结构、数据模型、UI 设计系统，避免后续频繁推翻重做。
4. 明确每个阶段要做什么、做到什么程度可以进入下一阶段。
5. 给 Codex + Figma 一套可执行的设计和改造指令。

核心原则：

```text
架构按未来设计，功能按当前落地。
UI 按 SaaS 统一，模块按阶段开放。
先固化内部流程，再对外商户化；先做提效系统，再做商户协作网络。
```

## 1. 产品重新定位

### 1.1 不再是“闲鱼助手”

当前项目最开始可能是围绕闲鱼业务产生的，包括询单、查档期、发档期、免押、发货、完结等。但未来产品不能继续被“闲鱼”这个渠道限制。

未来系统的核心不应该是：

```text
闲鱼自动回复工具
```

而应该是：

```text
租赁商户运营标准化系统
```

闲鱼、小红书、抖音、私域、线下、商户协作订单，都应该只是订单来源之一。

建议系统里统一使用：

```text
渠道 Channel = 闲鱼 / 小红书 / 抖音 / 私域 / 手动录入 / 线下 / 商户协作 / 其他
```

也就是说：

```text
闲鱼 = 渠道之一
不是产品核心
```

---

### 1.2 新产品定位

建议对外定位为：

```text
小狗相机助手商户版
面向相机、摄影器材、数码设备租赁商家的订单、档期、设备、物流、免押和经营数据管理系统。
```

更简短的表达：

```text
小狗相机助手商户版 = 租赁商家的运营管理系统。
```

第一阶段主要服务：

- 相机租赁商家
- 运动相机租赁商家
- CCD / 卡片机租赁商家
- 稳定器 / 无人机 / 摄影器材租赁商家
- 演唱会 / 旅游拍摄设备租赁商家

后续可以扩展到：

- 其他数码设备租赁
- 户外装备租赁
- 演出设备租赁
- 服装道具租赁
- 其他高频租赁品类

但第一版不要写得过于泛化。先聚焦影像设备租赁，产品才有清晰场景。

---

## 2. 最终方向与阶段关系

### 2.1 近期目标：内部提效系统

近期最重要的目标不是平台交易，而是把当前业务流程做成一个标准化系统。

目标是：

```text
让新员工也能快速上手，不需要靠老板口头教，也能完成询单、查档期、创建订单、发货、提醒归还、完结等基础流程。
```

近期价值：

- 降低新人培训成本
- 减少漏发货、漏提醒、漏完结
- 提高查档期和发货效率
- 让订单、设备、档期、客户和报表沉淀为数据
- 为后续多员工、多门店、多商户使用打基础

---

### 2.2 中期目标：商户 SaaS 系统

当系统能稳定支持一个商户内部使用后，再升级为多个商户可使用的 SaaS。

中期目标：

- 支持登录注册
- 支持商户账号
- 支持门店 / 仓库
- 支持员工权限
- 支持多商户数据隔离
- 支持商户初始化引导
- 支持套餐和收费能力
- 支持外部商户试用

---

### 2.3 长期目标：商户协作网络

当商户都在系统里管理设备、档期和订单后，才适合逐步做商户之间的协作。

长期可以扩展：

- 调货需求
- 可供设备发布
- 代发协作
- 商户报价
- 二手设备交易
- 商户信用
- 分润结算
- 纠纷处理
- 平台担保
- 行业数据分析

长期方向可以理解为：

```text
租赁商户 SaaS 系统 + 商户协作交易网络
```

但当前阶段不要急着做交易闭环。先把商户内部系统做好。

---

## 3. 标准业务流程设计

系统要围绕一条标准租赁流程设计。

建议主流程固定为：

```text
询单
→ 查档期
→ 报价 / 发档期
→ 创建订单
→ 创建免押 / 收押金
→ 确认发货
→ 物流跟踪
→ 租期中提醒
→ 到期提醒
→ 归还验机
→ 完结订单
→ 财务统计
→ 数据分析
```

### 3.1 每一步的系统价值

| 流程节点 | 系统要解决的问题 | 关键功能 |
|---|---|---|
| 询单 | 客户需求沉淀，不靠聊天记录记忆 | 询单记录、客户需求、渠道来源 |
| 查档期 | 快速判断某型号某时间是否可租 | 型号档期、设备占用、余量预警 |
| 报价 / 发档期 | 快速回复客户，减少重复话术 | 报价模板、档期模板、价格策略 |
| 创建订单 | 将客户需求转成正式履约记录 | 订单、客户、设备、租期、金额 |
| 免押 / 押金 | 规范押金与免押申请 | 免押记录、押金状态、审核状态 |
| 发货 | 降低漏发货风险 | 发货任务、物流单号、发货提醒 |
| 物流跟踪 | 知道设备是否签收 | 物流状态、签收时间、异常提醒 |
| 租期中提醒 | 减少客户忘记归还 | 租期提醒、归还提醒、续租提醒 |
| 归还验机 | 规范设备回收流程 | 验机清单、配件核对、损坏记录 |
| 完结订单 | 订单闭环，财务可统计 | 完结状态、押金释放、收入确认 |
| 数据分析 | 判断哪些型号赚钱、哪些流程低效 | 报表、ROI、渠道分析、设备利用率 |

---

### 3.2 订单状态建议

订单状态不要随意增加，建议先固定一套：

```text
草稿
待确认
待免押 / 待押金
待发货
已发货
运输中
已签收
租期中
待归还
归还中
待验机
已完结
已取消
异常
逾期
```

前端展示可以简化为：

```text
待发货
进行中
待归还
逾期
已完结
异常
```

系统内部状态可以更细，用户界面不必全部露出。

---

### 3.3 设备状态建议

设备状态建议固定为：

```text
空闲
已预订
待发货
运输中
在租
归还中
待验机
清洁中
缺件
维修中
不可用
已售出
报废
```

档期页面可以用更简化的状态：

```text
空闲
在租
发货
归还
清洁/缺件
余量不足
满租
```

```text
1. 商户端系统
2. 平台运营后台
3. 商户协作市场
```

这是最核心的系统，给租赁商家日常使用。

包含：

- 订单中心

- 财务结算

这是你作为平台方未来要用的后台。

包含：

### 4.3 商户协作市场

## 5. 手机端与 PC 端分工

### 5.1 手机端定位

手机端不是 PC 后台缩小版，而是：

```text
移动工作台
```

适合做：

- 看今日待办
- 快速查档期
- 快速创建订单
- 确认发货
- 查看物流
- 提醒归还
- 审核免押
- 扫码处理设备
- 查看设备简要状态
- 处理异常提醒

手机端底部导航建议固定为：

```text
工作台 / 订单 / 档期 / 设备 / 我的
```

说明：

- 免押不长期占底部导航，放在工作台快捷入口和订单详情中。
- 报表、设置、商户协作等放在“我的 / 更多”或 PC 端。
- 手机端重点是高频轻操作，不做复杂表格。

---

### 5.2 PC 端定位

PC 端是：

```text
完整管理后台
```

适合做：

- 批量订单处理
- 大量档期查看
- 设备管理
- 客户管理
- 报表分析
- 财务结算
- 权限设置
- 数据导入导出
- 多门店配置
- 商户协作管理

PC 端左侧导航建议固定为：

```text
工作台
询单中心
订单中心
档期中心
设备中心
客户中心
免押管理
物流发货
财务结算
报表中心
设置中心
```

未来扩展：

```text
商户协作
调货大厅
代发中心
二手设备市场
```

## 6. 数据模型基础设计

### 6.1 必须提前支持多商户

即使现在只有自己一个商户，也要从数据结构上为多商户使用做准备。

所有核心业务表都建议预留：

```text
merchant_id
store_id
created_by
updated_by
created_at
updated_at
deleted_at / is_deleted
```

否则后期从单商户改多商户会非常痛苦。

---

### 6.2 核心实体建议

#### 账号与商户

```text
User 用户
Merchant 商户
Store 门店
Warehouse 仓库
Staff 员工
Role 角色
Permission 权限
OperationLog 操作日志
```

#### 业务流程

```text
Channel 渠道
Customer 客户
Inquiry 询单
Quote 报价 / 档期回复
Order 订单
OrderItem 订单明细
Deposit 押金 / 免押
Shipment 物流
ReturnInspection 归还验机
Settlement 结算
FinanceRecord 财务流水
```

#### 设备与档期

```text
DeviceCategory 设备分类
DeviceModel 设备型号
DeviceUnit 单台设备
Accessory 配件
DeviceStatusLog 设备状态日志
Schedule 档期
MaintenanceRecord 维修 / 清洁记录
```

#### 报表与配置

```text
ReportSnapshot 报表快照

MerchantConfig 商户配置
NotificationSetting 通知设置
PrintTemplate 打印模板

```

#### 未来协作

```text

MerchantQuote 商户报价
MerchantCredit 商户信用

```

## 7. 角色与权限框架

为了让新员工快速上手，必须提前规划角色权限。

建议第一版角色如下：

| 角色 | 核心权限 | 备注 |
|---|---|---|
| 老板 / 超级管理员 | 全部权限、财务、员工、设置 | 最高权限 |
| 店长 | 订单、设备、档期、员工日常管理 | 负责门店运营 |
| 运营 | 询单、报价、查档期、创建订单、客户沟通 | 高频业务角色 |
| 发货员 | 发货、物流、归还验机、设备状态 | 可限制看财务 |
| 财务 | 收入、押金、结算、报表 | 不一定能改订单 |
| 临时员工 | 只看分配任务 | 用于兼职或新人 |

权限模型建议采用：

```text
角色 Role → 权限 Permission → 菜单 / 按钮 / 数据范围
```

第一版可以简单做，但数据库和前端结构要预留。

## 8. 阶段路线图总览

建议分 7 个阶段推进。

| 阶段 | 阶段名称 | 核心目标 | 是否当前重点 |
|---|---|---|---|
| Phase 0 | 当前系统盘点与方向冻结 | 明确不再围绕闲鱼，冻结产品主干 | 是 |
| Phase 1 | 设计系统与 UI 底座 | 固定 UI 语言和页面骨架 | 是 |

| Phase 5 | 对外商户试用版 | 支持外部商户使用 | 中期 |
| Phase 6 | 商户协作信息撮合 | 调货、代发、可供设备的轻量 MVP | 后期 |
| Phase 7 | 交易闭环与平台化 | 分润、担保、二手交易、信用体系 | 长期 |

# 9. Phase 0：当前系统盘点与方向冻结

## 9.1 阶段目标

把当前项目从“闲鱼助手”思维中抽离出来，明确新定位，盘点当前已有功能、页面、数据结构和技术债，确保后续改造有明确边界。

## 9.2 本阶段要做什么

1. 梳理当前项目所有页面。
2. 梳理当前所有业务功能。
3. 梳理当前数据库表结构。
4. 梳理哪些功能与“闲鱼”强绑定。
5. 判断哪些功能保留、哪些废弃、哪些重构。
6. 冻结新的产品定位和导航结构。
7. 建立新项目改造分支。
8. 备份当前可用版本。

## 9.3 当前功能处理建议

| 当前功能 | 处理方式 | 原因 |
|---|---|---|
| 闲鱼自动回复 | 可以弱化或舍弃 | 与未来定位关系不强 |
| 查档期 | 保留并重构 | 租赁核心能力 |
| 发档期 | 保留并泛化为报价 / 档期回复 | 不再只服务闲鱼 |
| 创建免押 | 保留 | 履约关键流程 |
| 发货 | 保留 | 履约核心 |
| 完结订单 | 保留 | 订单闭环 |
| 数据汇总 | 保留并升级为报表 | SaaS 价值点 |
| 配置项 | 保留并迁移到商家中心 | 多商户需要配置体系 |

## 9.4 阶段交付物

- 当前功能盘点表
- 当前页面路由图
- 当前数据库表结构说明
- 保留 / 废弃 / 重构功能清单
- 新产品定位确认
- 新导航结构确认
- 改造分支创建完成

## 9.5 进入下一阶段的标志

满足以下条件，才进入 Phase 1：

```text
1. 已确认产品不再以闲鱼为核心。
2. 已确认手机端主导航：工作台 / 订单 / 档期 / 设备 / 我的。
3. 已确认 PC 端主导航方向。
4. 已完成当前项目功能和页面盘点。
5. 已明确哪些功能要保留、舍弃、重构。
6. 已创建安全改造分支，当前可用版本可回退。
```

## 9.6 Phase 0 需求文档摘要

### 背景

当前系统来源于内部租赁业务提效工具，但未来要升级为租赁商户运营系统，因此需要先完成方向冻结和项目盘点。

### 用户

- 当前用户：老板 / 内部运营人员
- 未来用户：租赁商户老板、店长、运营、发货员、财务

### 需求范围

- 不新增大功能
- 不大规模改 UI
- 只做分析、盘点、冻结方向和改造计划

### 验收标准

- 能清楚说明当前项目有哪些功能
- 能清楚说明哪些功能进入新系统
- 能清楚说明后续改造路线

# 10. Phase 1：设计系统与 UI 底座

## 10.1 阶段目标

固定产品的 UI 设计语言，建立可复用组件，避免后续每次页面都重新设计、重新改代码。

本阶段重点不是做大量新功能，而是建立：

```text
Design System + App Shell + 基础组件库
```

## 10.2 本阶段要做什么

### 设计侧

1. 在 Figma 中创建设计系统文件。
2. 定义颜色、字体、间距、圆角、阴影。
3. 定义按钮、输入框、标签、卡片、列表、表格等组件。
4. 定义手机端页面框架。
5. 定义 PC 端页面框架。
6. 输出核心页面高保真设计：
   - 登录页
   - 工作台
   - 订单中心
   - 档期概览
   - 设备中心
   - 商家中心

### 开发侧

1. 创建前端设计 tokens。
2. 抽取基础组件。
3. 重构全局 AppShell。
4. 统一按钮、卡片、标签、表格等样式。
5. 保持现有功能不被破坏。

## 10.3 UI 风格方向

关键词：

```text
专业、可信、克制、清爽、高效、B 端 SaaS 感
```

不建议：

```text
过度可爱
大面积渐变
过多阴影
```

# 11. Phase 2：登录、商户体系与数据隔离

## 11.1 阶段目标

让系统从“单人内部工具”变成“可以承载多个商户的系统”。

这是 SaaS 化的地基。

## 11.2 本阶段要做什么

1. 增加登录系统。
2. 增加用户表 User。
3. 增加商户表 Merchant。
4. 增加门店 Store 和仓库 Warehouse。
5. 增加员工 Staff、角色 Role、权限 Permission。
6. 所有核心业务数据增加 merchant_id。
7. 需要按门店区分的数据增加 store_id。
8. 增加当前商户 / 当前门店上下文。
9. 增加商家中心基础配置。
10. 增加操作日志。
11. 做现有数据迁移方案。

## 11.3 登录方式建议

第一版支持：

- 手机号验证码登录
- 密码登录
- 管理员创建员工账号

后续支持：

- 微信登录
- 企业微信
- 多设备登录管理
- 登录安全提醒

## 11.4 商户初始化流程

新商户第一次进入系统时，引导完成：

```text
Step 1：创建商户
Step 2：创建门店 / 仓库
Step 3：导入设备
Step 4：邀请员工
Step 5：开始创建订单
```

## 11.5 数据隔离原则

所有查询必须带上：

```text
merchant_id
```

涉及门店的数据带上：

```text
store_id
```

普通员工只能访问自己商户的数据，不允许跨商户访问。

## 11.6 阶段交付物

- 登录页
- 注册 / 创建商户流程
- 商户中心
- 员工与权限基础版
- 多租户数据字段
- 数据迁移脚本
- 操作日志
- 当前商户 / 门店切换

## 11.7 进入下一阶段的标志

满足以下条件，才进入 Phase 3：

```text
1. 用户可以登录系统。
2. 系统支持至少一个商户实体。
3. 所有核心业务数据已绑定 merchant_id。
4. 门店相关数据已支持 store_id。
5. 员工可以归属于某个商户。
6. 至少支持老板 / 运营 / 发货员三个基础角色。
7. 旧数据已能迁移到默认商户下。
8. 不同商户的数据在测试环境中无法互相访问。
```

## 11.8 Phase 2 需求文档摘要

### 背景

系统要从内部工具升级为商户 SaaS，必须支持账号、商户、门店和数据隔离。

### 目标

建立 SaaS 地基，允许未来多个商户使用系统。

### 功能范围

- 登录
- 商户
- 门店
- 员工
- 权限
- 数据隔离
- 数据迁移

### 验收标准

- 可登录
- 可创建商户
- 可切换门店
- 数据按商户隔离

- 免押

- 归还
- 完结

### 验收标准

- 标准订单可完整流转
- 设备状态与档期正确变化
- 工作台能显示待办

# 15. Phase 6：商户协作信息撮合 MVP

## 15.1 阶段目标

在已有商户使用系统的基础上，先做轻量级的商户协作信息撮合。

不要一开始做支付、担保、分润和纠纷闭环。

第一版只做：

```text
发布需求 → 商户看到 → 联系 / 报价 → 记录结果
```

## 15.2 可做的协作类型

### 调货需求

商户缺设备时发布：

- 需要型号
- 数量
- 时间
- 城市
- 预算
- 履约方式
- 联系方式

### 可供设备

设备空闲商户发布：

- 可供型号
- 数量
- 可用时间
- 结算价
- 是否支持代发
- 是否支持同城调货

### 代发需求

有流量但没设备的商户发布：

- 客户订单需求
- 设备型号
- 时间
- 客户城市
- 分润方式
- 由谁发货

### 二手设备信息

商户发布：

- 出售设备
- 求购设备
- 批量设备包
- 成色
- 价格
- 所在城市

## 15.3 阶段交付物

- 商户协作入口
- 需求发布
- 可供设备发布
- 需求列表
- 商户联系 / 报价
- 简单信用展示
- 成交结果记录

## 15.4 进入下一阶段的标志

满足以下条件，才进入 Phase 7：

```text
1. 有真实商户发布调货或代发需求。
2. 有其他商户响应。
3. 至少形成若干次真实线下成交或协作。
4. 发现高频纠纷点和责任边界问题。
5. 商户愿意为成交、曝光或效率付费。
6. 平台已经知道哪些交易规则必须系统化。
```

## 15.5 Phase 6 需求文档摘要

### 背景

商户之间已有微信群调货和代发需求，但目前效率低、记录散、信用弱。

### 目标

先把商户协作从微信群信息流转移到系统记录中。

### 功能范围

- 发布需求
- 发布可供设备
- 浏览需求
- 联系商户
- 记录结果

### 非目标

- 不做支付闭环
- 不做平台担保
- 不做复杂分润
- 不做纠纷仲裁系统

### 验收标准

- 有真实商户使用
- 有真实需求和响应
- 能验证协作价值

# 24. 最终结论

你当前最正确的方向不是继续做“闲鱼助手”，也不是马上做“大而全平台”。

最适合的路线是：

```text
第一步：把当前业务提效工具升级为租赁商户运营标准化系统。
第二步：补齐登录、商户、门店、员工、权限和数据隔离，形成 SaaS 地基。
第三步：用统一设计系统重构手机端和 PC 端核心页面。
第四步：通过 SOP 和角色工作台，让新员工可以快速上手。
第五步：小范围开放给外部商户试用。
第六步：在真实商户需求基础上，再做调货、代发、二手交易等商户协作。
```

一句话原则：

```text
先标准化自己的业务，再 SaaS 化给商户；先做内部提效，再做商户协作；先固定架构和设计系统，再持续扩展功能。
```
