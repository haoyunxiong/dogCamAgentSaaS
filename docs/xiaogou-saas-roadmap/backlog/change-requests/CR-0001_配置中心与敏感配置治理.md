# CR-0001 配置中心与敏感配置治理

## 1. 需求背景

当前项目中可能存在配置散落问题：

- JS / Python 文件中有 token、cookie、password、mysql 等变量或配置读取逻辑；
- 本地环境文件、MySQL 连接文件、日志、数据备份不应进入 Git；
- 部分配置可能已经存在 SQLite / MySQL；
- 后续项目要支持多商户、多门店、多员工，因此商户配置需要按 merchant_id / store_id 隔离；
- 敏感凭证不能写死在代码里，也不能提交到 Git。

## 2. 需求目标

1. 审计当前所有重要配置来源；
2. 区分系统默认配置、本地运行配置、商户业务配置、敏感凭证；
3. 判断哪些配置当前在文件中，哪些在 DB 中；
4. 判断哪些配置未来应进入 MySQL；
5. 判断哪些敏感凭证需要本地加密或服务端加密；
6. 建立 ConfigService / CredentialService 的后续改造方向；
7. 形成配置治理文档，不在当前阶段贸然重构。

## 4. 本次明确不做

- 不在首次 Git baseline 阶段重构配置系统；
- 不直接新增数据库表；
- 不直接迁移配置数据；
- 不直接删除旧配置；
- 不修改业务行为；
- 不把敏感值打印到日志或文档中。

## 5. 配置分类规则

### 5.1 系统默认配置

可以保留在代码配置文件中，可以提交 Git。

示例：默认分页、默认状态、默认枚举、默认 UI 规则。

### 5.2 本地运行配置

不提交 Git。保存在 local.env、mysql-connection.json、本地配置文件中。

示例：本地数据库地址、本地端口、本地调试开关。

### 5.3 商户业务配置

未来应存储在 DB / MySQL 中，并按 merchant_id / store_id 隔离。

示例：商户信息、门店配置、物流配置、免押规则、定价规则、打印模板、员工权限。

### 5.4 敏感凭证

不能写死在代码中，不能提交 Git。未来通过 CredentialService 读取。

示例：API Key、Cookie、Token、数据库密码、顺丰密钥、免押接口密钥、AI 模型 Key。
