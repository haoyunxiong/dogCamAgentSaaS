# 小狗相机助手商户版：Codex 多会话创建、使用与提问手册

> 适用项目：小狗相机助手商户版 / 租赁商户运营标准化系统  
> 适用场景：你希望用 Codex 分阶段推进项目、操作 Figma、改造代码、做验收，同时避免 Codex 反复读取大文档、被旧“闲鱼助手”逻辑干扰、跨阶段乱改。

---

## 0. 先给结论

你可以提前创建多个 Codex 会话，每个会话只负责一种流程。推荐创建 6 个固定会话：

| 会话 | 用途 | 是否改代码 | 是否操作 Figma | 是否改数据库 |
|---|---|---:|---:|---:|
| 01 总控台 / 阶段调度 | 判断当前阶段、告诉你去哪个会话、生成可复制 Prompt | 否 | 否 | 否 |
| 02 Figma 设计系统 | 设计系统、组件库、移动端/PC 原型 | 否 | 是 | 否 |
| 03 前端 UI 与交互 | 前端组件、页面、交互、样式改造 | 是 | 否 | 否 |
| 04 SaaS 底座 | 登录、商户、门店、员工、权限、数据隔离 | 是 | 否 | 是，但必须先计划和备份 |
| 05 业务流程与 SOP | 订单流、档期流、发货、归还、员工上手流程 | 是 | 可读设计，不主设计 | 谨慎，按阶段决定 |
| 06 验收审查 | 检查阶段成果、测试、是否允许进入下一阶段 | 原则上否 | 否 | 否 |

核心原则：

```text
会话 = 工作台
文档 = 项目真相
Git = 代码真相
Figma = 设计真相
阶段状态表 = 进度真相
```

不要依赖某个 Codex 会话的历史记忆作为唯一依据。每次真正执行前，都要让它重新读取：

```text
AGENTS.md
README_如何使用这套文档.md
docs/xiaogou-saas-roadmap/00_README_总控入口.md
docs/xiaogou-saas-roadmap/05_PHASE_STATUS_阶段状态表.md
当前阶段文件夹
```

## 1. 总控台能不能每次直接告诉你该去哪个会话、用什么 Prompt？

可以，而且强烈建议这么做。

你可以把“总控台”当成一个**调度会话**，它不负责直接改代码，也不直接操作 Figma。它只负责：

1. 读取当前阶段状态；
2. 判断应该使用哪个会话；
3. 判断是否需要新建执行会话；
4. 给你生成一段可复制给目标会话的 Prompt；
5. 告诉你执行完成后要把什么结果回传给总控台；
6. 检查是否满足进入下一阶段的条件。

总控台每次应该输出一种固定格式，叫：

```text
【总控台调度单】
```

调度单模板如下：

```text
【总控台调度单】

1. 当前阶段：
例如：Phase 02 - Design System

2. 本次目标：
例如：创建 Figma 设计系统，不写代码。

3. 推荐使用的会话：
例如：02 Figma 设计系统

4. 是否建议新建会话：
例如：否，继续使用固定 Figma 设计会话。

5. 本次允许做：
- 读取当前阶段文档
- 使用 Figma MCP
- 创建 Design System 页面
- 创建组件和变量

6. 本次禁止做：
- 不改项目代码
- 不改数据库
- 不做商户协作功能
- 不生成 PNG 作为最终稿

7. 请复制到目标会话的 Prompt：
【这里由总控台生成完整 Prompt】

8. 目标会话完成后，需要回传给总控台：
- 做了什么
- 改了哪些文件 / Figma 页面
- 哪些未完成
- 如何验证
- 是否满足验收标准
- 是否建议进入下一阶段
```

## 2. 你要先创建哪些 Codex 会话

建议你在 Codex 里提前创建以下 6 个会话，并固定名称。

### 2.1 会话 01：总控台 / 阶段调度

推荐名称：

```text
小狗相机助手 - 总控台与阶段调度
```

职责：

```text
只负责阶段判断、任务调度、生成目标会话 Prompt、判断是否进入下一阶段。
不直接改代码、不操作 Figma、不执行数据库迁移。
```

首次创建时发送：

```text
你是“小狗相机助手商户版”项目的总控台与阶段调度代理。

项目新定位：
本项目已经从“闲鱼助手”升级为“租赁商户运营标准化系统”。
闲鱼不再是产品核心，只能作为订单来源渠道之一。
当前阶段目标以 docs/xiaogou-saas-roadmap/05_PHASE_STATUS_阶段状态表.md 为准。

你的职责：
1. 读取 AGENTS.md；
2. 读取 docs/xiaogou-saas-roadmap/00_README_总控入口.md；
3. 读取 docs/xiaogou-saas-roadmap/05_PHASE_STATUS_阶段状态表.md；
4. 判断当前应该执行哪个阶段；
5. 判断应该使用哪个 Codex 会话；
6. 为目标会话生成可直接复制的 Prompt；
7. 检查阶段验收标准；
8. 防止跨阶段开发。

你不能做：
1. 不直接修改代码；
2. 不直接操作 Figma；
3. 不执行数据库迁移；
4. 不实现功能；
5. 不把历史会话记忆当成唯一依据。

每次我来找你时，请输出【总控台调度单】，告诉我：
1. 当前阶段是什么；
2. 应该去哪个会话；
3. 是否需要新建会话；
4. 复制给目标会话的完整 Prompt；
5. 目标会话完成后应回传什么内容。

现在请先不要修改任何文件。请只读取项目规划文档，并告诉我你理解的当前项目执行规则。
```

日常使用时发送：

```text
请作为总控台读取当前阶段状态表，判断我现在应该继续哪个阶段、使用哪个会话，并生成一段可以直接复制到目标会话的 Prompt。

要求：
1. 不要修改代码；
2. 不要操作 Figma；
3. 不要执行数据库迁移；
4. 只输出【总控台调度单】。
```

### 2.2 会话 02：Figma 设计系统

推荐名称：

```text
小狗相机助手 - Figma 设计系统
```

```text

不改项目代码，不改数据库。
```

首次创建时发送：

```text
你是“小狗相机助手商户版”的 Figma 设计系统代理。

小狗相机助手商户版是面向租赁商家的运营标准化系统，帮助商户管理订单、档期、设备、物流、免押、客户、员工和经营数据。

重要原则：
1. 闲鱼不再是产品核心，只是订单来源渠道之一；
2. 当前先做商户内部提效系统，不做商户调货、代发、二手交易闭环；
3. UI 要体现 B 端 SaaS 气质：专业、可信、克制、清爽、高级、效率优先；

6. 每次先进入计划模式，我确认后再操作 Figma。

你的职责：
1. 使用 Figma MCP；
2. 创建 Figma 原生可编辑图层；
3. 创建 Design System；
4. 创建组件库；
5. 创建移动端核心页面；
6. 创建 PC 端后台页面；
7. 所有页面复用统一组件和变量。

你不能做：
1. 不改项目代码；
2. 不执行数据库迁移；
3. 不生成单张 PNG 冒充设计稿；
4. 不跨阶段设计未来交易功能；
5. 不随意改变已经确认的设计系统。

以后我会从总控台复制 Prompt 给你。收到任务后，请先输出 Figma 执行计划，等待我确认。
```

## 3. 每次执行的标准流程

以后每次你推进项目，都按下面 8 步走。

### Step 1：先问总控台

发送：

```text
请读取当前阶段状态表，判断我现在应该执行哪个阶段、使用哪个会话，并生成给目标会话的完整 Prompt。
只输出【总控台调度单】，不要修改代码。
```

### Step 2：总控台输出调度单

你看它推荐哪个会话，例如：

```text
推荐使用：02 Figma 设计系统
复制以下 Prompt 到目标会话：...
```

### Step 3：复制 Prompt 到目标会话

不要自己重新组织语言，尽量复制总控台给你的完整 Prompt。

### Step 4：目标会话先输出计划

目标会话必须先输出：

```text
我要做什么
不做什么
准备改什么
风险是什么
如何验证
```

### Step 5：你确认计划

确认时发送：

```text
计划确认，可以执行。
严格按本次计划执行，不要扩大范围。
执行完成后输出变更清单、验证方式和回传给总控台的报告。
```

### Step 6：目标会话执行

执行完后，它要输出：

```text
完成了什么
改了哪些文件 / Figma 页面
没有做什么
如何验证
是否满足验收标准
需要回传给总控台的总结
```

### Step 7：把执行结果回传给总控台

发送给总控台：

```text
这是目标会话的执行结果，请判断是否满足当前阶段要求，是否需要继续修复，是否可以进入验收。

【粘贴执行结果】
```

### Step 8：让验收会话检查

## 12. 你最应该记住的 5 条规则

```text
1. 每次先问总控台。
2. 总控台不执行，只调度。
3. 目标会话先计划，你确认后再执行。
4. 执行结果回传总控台，再决定是否验收。
5. 验收通过后，才进入下一阶段。
```

## 13. 最推荐的日常工作流

```text
你 → 总控台：我现在要推进项目，下一步做什么？
总控台 → 你：去 Figma 会话，复制这段 Prompt。
你 → Figma 会话：粘贴 Prompt。
Figma 会话 → 你：输出计划。
你 → Figma 会话：确认执行。
Figma 会话 → 你：完成并输出结果。
你 → 总控台：粘贴执行结果。
总控台 → 你：建议进入验收会话。
你 → 验收会话：粘贴验收 Prompt。
验收会话 → 你：输出是否通过。
你 → 总控台：粘贴验收报告。
总控台 → 你：判断是否进入下一阶段。
```

## 14. 这套多会话体系的最终目的

它的目的不是把事情变复杂，而是让 Codex 不再乱。

你的项目未来会越来越复杂：

```text
设计系统
手机端
PC 端
登录
商户
门店
员工
权限
数据隔离
订单
档期
设备
免押
报表
SOP
未来商户协作
```

如果没有总控和阶段化，Codex 很容易：

```text
一边做 Figma
一边改数据库
一边重构 UI
一边又加未来功能
最后全乱
```

所以正确方式是：

```text
总控台管方向
专项会话管执行
验收会话管质量
阶段文档管边界
Git / Figma / 数据库管真实状态
```

这样你才能长期、稳定、可控地把当前项目从内部助手改造成真正的租赁商户 SaaS 系统。
